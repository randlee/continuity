# Continuity Python Implementation — Issue Report

**Date:** 2027-07-08 (updated 2027-07-09 with per-repo timer impact analysis)  
**Scope:** Python implementation vs. `docs/architecture.md`, `docs/requirements.md`, `docs/requirements-atm.md`, `docs/sprint-plan-http-rpc-migration.md`, `docs/sprint-plan-per-repo-timers.md`  
**Status:** Open  
**Test baseline:** 437 tests passing (`python3 -m pytest python/tests/ -v`)

---

## Summary

The Python codebase implements most architectural modules as a well-tested library, but several documented contracts are incomplete or inconsistent. The highest-impact gaps are operability (no `ci` command driver), broken EMA persistence (Phase 5), incomplete ATM notification delivery, and missing end-to-end verification.

### `cli` module vs. `ci` command

Requirements define two related but distinct surfaces:

| Surface | Requirements ref | Status |
|---------|------------------|--------|
| **`cli` module** — thin HTTP client implementation | §4.4 (`python/cli/`) | **Mostly done** — `daemon_cmd.py`, `http_client.py`, `cmd_*` functions, unit tests |
| **`ci` command** — user-facing invocation (`ci daemon`, `ci status`, …) | §9 CLI Command Summary | **Missing** — no `ci` executable wires subcommands to the `cli` module |

The gap is not the `cli` module itself; it is the **`ci` entry point** that requirements §9 specifies as how users run the tool.

Issues are ordered by priority. Each section includes implementation tasks and a test plan with corner cases.

## Issue 1 — No `ci` Command Entry Point (Critical)

**Refs:** FR-38–FR-44, FR-33, requirements §4.4, §9, architecture §9, §12  
**Severity:** Critical — documented product interface is not runnable

### Problem

The **`cli` module** (requirements §4.4) is implemented: `python/cli/daemon_cmd.py` exposes `cmd_status()`, `cmd_log()`, `cmd_poll()`, etc., and `python/cli/http_client.py` talks to the daemon HTTP RPC. `daemon.py` has `Daemon.start()`.

What is missing is the **`ci` command** (requirements §9): a user-invokable entry point that maps `ci <subcommand>` to those `cmd_*` functions. Users cannot run `ci daemon`, `ci status`, `ci log`, etc. The `conftest.py` `daemon` fixture is a stub (`TODO: replace with actual daemon start`).

Note: the sprint plan suggested `python/continuity` as a filename, but requirements consistently name the **command** `ci`, not `continuity`.
